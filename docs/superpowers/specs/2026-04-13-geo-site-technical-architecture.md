# GEO 官网整站技术架构文档

## 1. 文档目的

本文档定义 GEO 官网整站 MVP 的技术架构方案，覆盖系统拆分、前后端边界、身份认证、数据域划分、接口访问路径、埋点与转化漏斗、部署方式、安全边界和扩展方向。

本文档聚焦整站层面的架构设计，不展开测试功能内部的详细算法、单次测试统计逻辑或字段级实现细节。测试功能内部设计以《GEO 测试功能技术设计文档》为准。

## 2. 架构目标

整站技术架构需满足以下目标：

- 支撑官网首页、测试页、结果页三段式产品结构
- 支撑注册留资、免费测试、结果展示和后续联系转化
- 保持前端站点与测试业务逻辑分离
- 以 Supabase 作为认证与数据中台
- 将埋点与转化漏斗纳入正式技术范围
- 在保证 MVP 实现效率的前提下，为后续扩展预留边界

## 3. 架构原则

- 首期采用“具体技术栈落地 + 关键层次抽象”的方式设计
- 前端站点与业务后端分开部署、分开演进
- 认证能力由 Supabase Auth 承担
- 核心业务规则由独立后端统一控制
- 测试结果与累计指标以后端返回为准
- 埋点体系独立纳入正式范围，不作为附属能力
- 内容配置、业务数据和埋点数据分域管理
- 所有敏感配置均由服务端环境管理，不下发至前端

## 4. 技术选型基线

首期建议基于以下技术栈实施：

- 前端站点：Next.js（TypeScript）
- 前端部署：Vercel
- 业务后端：Python + FastAPI
- 后端部署：阿里云函数计算或轻量应用服务器
- 身份认证：Supabase Auth（邮箱 + 密码模式）
- 业务数据库：Supabase Database（PostgreSQL）
- 邮件通知：阿里云邮件推送或 Resend 等轻量邮件服务
- 埋点分析：轻量事件采集 + 转化漏斗分析能力（建议 Google Analytics 4 或 GrowingIO）

说明：

- 本文档主文按上述栈写法展开
- 关键边界按抽象层设计，未来可替换具体实现
- 后端选择 Python + FastAPI 的原因：LLM 调用生态成熟，后续扩展 NLP 能力方便
- 后端不部署在 Vercel 上，因模型调用耗时可能超过 Vercel Serverless 默认超时限制

## 5. 系统总体结构

首期整站建议拆分为四个系统域：

1. 官网前端站点
2. 独立业务后端服务
3. Supabase 中台
4. 埋点与分析层

### 5.1 官网前端站点

职责：

- 渲染首页、测试页、结果页
- 承担用户交互与页面状态管理
- 承接注册弹窗与登录态展示
- 保存测试表单前端暂存内容
- 展示总体累计数据和单次测试结果
- 上报前端曝光与交互埋点

不承担：

- 测试规则计算
- 免费次数裁决
- 公司名匹配
- 模型调用
- 累计指标聚合

### 5.2 独立业务后端服务

职责：

- 提供整站业务 API
- 处理测试发起与结果返回
- 管理免费次数、测试资格、业务规则
- 执行 Prompt 生成、模型调用、结果审查与聚合更新
- 对前端输出结构化业务结果
- 承接后续转化动作，如联系专属 AI 营销团队

不承担：

- 页面渲染
- 前端视觉状态控制

### 5.3 Supabase 中台

职责：

- 提供邮箱 + 密码注册与登录认证能力
- 存储业务用户资料
- 存储测试记录、用户累计指标与模板配置
- 作为整站认证与数据基础设施

不承担：

- 核心业务流程编排
- 业务后端的模型调用逻辑
- 前端直接可见的完整业务规则判断

### 5.4 埋点与分析层

职责：

- 记录关键页面曝光、CTA 点击、表单行为、注册完成、测试完成和后续联系事件
- 还原转化漏斗
- 支撑增长分析与页面优化

要求：

- 不阻塞主链路
- 与业务事实数据可关联但不相互替代

## 6. 整站端到端数据流

### 6.1 首页访问链路

用户进入首页后，前端站点完成：

- 页面内容渲染
- 基础身份状态感知
- 首页曝光与首屏曝光埋点上报

首页以流量承接和转化引导为主，不承担复杂业务计算。

### 6.2 首页进入测试页链路

当用户点击首页 CTA 后：

- 前端上报对应 CTA 点击事件
- 跳转至测试页
- 测试页加载时获取当前用户状态和总体累计数据

如用户未注册，则显示空状态或示例状态；如已注册，则显示真实累计值。

### 6.3 测试页填写与注册链路

测试页流程如下：

1. 顶部展示总体累计数据
2. 用户填写测试表单
3. 前端本地暂存表单内容
4. 用户点击 `查看AI曝光情况`
5. 系统在此时判断是否已注册
6. 若未注册，则拉起注册流程
7. 注册成功后恢复并回填此前填写内容
8. 系统继续执行测试请求

说明：

- 注册校验发生在提交测试动作时，而不是进入测试页时
- 该设计用于利用沉没成本提升注册转化率

### 6.4 测试执行与结果返回链路

测试请求发起后：

1. 前端调用业务后端测试接口
2. 后端校验用户身份、免费次数和请求参数
3. 后端执行测试链路并更新数据
4. 后端返回本次测试结果与最新累计指标
5. 前端展示结果页

### 6.5 结果页与后续转化链路

结果页承担三类职责：

- 展示顶部总体累计数据
- 展示本次测试结果明细
- 承接 `联系您的专属AI营销团队` 动作

结果页曝光与后续联系动作都应进入埋点漏斗。

## 7. 页面层与服务层边界

### 7.1 前端站点边界

前端负责：

- 页面展示
- 表单交互
- 前端状态管理
- 注册弹窗触发
- 本地暂存与回填
- 埋点触发

前端不负责：

- 免费次数判断
- 业务口径计算
- 测试结果聚合
- 测试资格判断

### 7.2 业务后端边界

后端负责：

- 提供用户上下文能力
- 提供测试执行能力
- 提供测试结果读取能力
- 提供配置下发能力
- 提供后续联系承接能力

后端统一管理业务规则，前端不绕过后端直接修改业务数据。

### 7.3 前端与后端协作原则

- 前端只消费结构化业务结果
- 后端只暴露与当前页面需求相关的稳定能力
- 前端不自行推导核心累计指标
- 所有测试结果与免费次数以后端确认为准

## 8. 整站数据域设计

整站数据建议分为以下四个域：

1. 身份与注册数据
2. 测试业务数据
3. 内容与配置数据
4. 埋点与分析数据

### 8.1 身份与注册数据

包含：

- Supabase Auth 身份
- 业务用户信息
- 注册时间与基础公司信息

说明：

- Auth 用户与业务用户资料逻辑上分层
- 业务逻辑应以业务用户资料和后端校验结果为准

### 8.2 测试业务数据

包含：

- 单次测试输入
- 单次测试结果
- 用户累计指标
- 免费测试次数
- 测试状态与失败信息

要求：

- 由后端统一维护
- 支持结果回溯

### 8.3 内容与配置数据

包含：

- 行业内容
- FAQ 内容
- 关于我们内容
- Prompt 模板配置
- 平台开关配置
- 免费测试次数等业务配置

要求：

- 与业务事实数据分离
- 尽量配置化管理

### 8.4 埋点与分析数据

包含：

- 页面曝光事件
- CTA 点击事件
- 表单填写与注册事件
- 测试完成与失败事件
- 后续联系意向事件

要求：

- 独立于业务事实数据存在
- 用于分析和漏斗优化，不直接替代业务记录

## 9. 整站接口与访问路径

### 9.1 前端的两条访问路径

前端站点保留两条明确访问路径：

1. 直接访问 Supabase Auth

用于邮箱注册、登录和认证状态承接

1. 访问独立业务后端 API

用于用户上下文、测试执行、测试结果、业务配置和后续联系动作

### 9.2 不建议的访问方式

不建议前端绕过业务后端直接读写核心业务表。

原因：

- 免费次数属于后端业务规则
- 测试指标属于后端聚合结果
- 测试执行涉及模型调用和计费成本
- 直接读写会破坏规则统一性

### 9.3 后端能力划分

后端建议至少暴露以下类型的能力：

- `User Context API`
- `Test Execution API`
- `Test Detail API`
- `Config API`
- `Conversion API`

说明：

- `User Context API` 服务于测试页和结果页顶部总体数据展示
- `Config API` 服务于平台开关、免费规则和轻量业务文案下发
- `Conversion API` 服务于"联系专属AI营销团队"动作，触发线索邮件通知

### 9.4 后台管理能力

首期需提供简易后台管理页面，供运营查看：

- 注册用户列表（邮箱、手机号、公司名、注册时间）
- 用户测试记录与累计指标
- 联系线索记录

说明：

- 首期可直接使用 Supabase Dashboard 查看数据库，或搭建一个简单的 admin 页面
- 不需要复杂的权限管理和操作能力

## 10. 埋点与转化漏斗架构

### 10.1 埋点目标

整站埋点用于：

- 还原从首页到结果页的转化链路
- 识别用户流失环节
- 评估 CTA、测试页、注册流程和结果页设计效果
- 支撑后续增长优化

### 10.2 主漏斗链路

首期漏斗主链路建议为：

`首页访问 -> CTA 点击 -> 进入测试页 -> 开始填写表单 -> 点击查看AI曝光情况 -> 拉起注册 -> 注册完成 -> 测试完成 -> 结果页曝光 -> 点击联系专属AI营销团队`

### 10.3 事件分层

埋点事件建议分为：

- 曝光事件
- 交互事件
- 转化事件
- 失败与流失事件

### 10.4 关键事件建议

首期至少包括以下核心事件：

曝光类：

- `landing_page_view`
- `hero_cta_exposed`
- `test_page_view`
- `result_page_view`
- `registration_modal_view`

交互类：

- `hero_cta_click`
- `industry_cta_click`
- `test_form_started`
- `test_form_submitted`
- `registration_modal_opened`
- `contact_ai_team_clicked`

转化类：

- `registration_completed`
- `test_execution_completed`
- `contact_ai_team_submitted`

失败类：

- `registration_failed`
- `test_execution_failed`
- `quota_exhausted_exposed`

### 10.5 前后端上报分工

前端适合上报：

- 页面曝光
- 按钮点击
- 表单开始填写
- 注册弹窗打开

后端适合上报：

- 测试真正发起
- 测试真正完成
- 测试失败
- 免费次数扣减成功
- 后续联系意向写入成功

原则：

- 用户行为事件以前端为准
- 业务成功事实以后端为准

## 11. 部署结构与环境划分

### 11.1 部署结构

首期建议采用以下部署结构：

- 前端站点独立部署在 Vercel
- 业务后端（Python + FastAPI）独立部署在阿里云函数计算或轻量应用服务器
- Supabase 提供邮箱认证与 PostgreSQL 数据库中台
- 邮件通知服务用于线索通知
- 埋点系统独立记录事件，不阻塞主链路

### 11.2 域名与访问边界

建议逻辑上区分两类入口：

1. 用户可见站点域名

用于承接首页、测试页和结果页

1. 后端 API 域名

用于前端访问业务接口

说明：

- 站点域名和 API 域名属于不同运行边界
- 后端 API 不直接作为用户访问入口使用

### 11.3 跨域与 CORS 配置

前端站点（Vercel）与业务后端（独立部署）属于不同域名，需要配置 CORS：

- 业务后端 FastAPI 需配置 CORS 中间件，允许前端站点域名的跨域请求
- 允许的 HTTP 方法：GET、POST、OPTIONS
- 允许携带认证信息（credentials）
- Supabase 客户端 SDK 的跨域由 Supabase 平台自行处理，无需额外配置

说明：

- 开发环境需允许 `localhost` 跨域
- 生产环境需严格限制为前端站点的正式域名

### 11.4 环境划分

建议逻辑上至少划分：

- 本地开发环境
- 测试/预发环境
- 生产环境

至少应隔离以下内容：

- Supabase 数据环境
- 后端环境变量
- 模型平台密钥
- 埋点事件环境标识
- 前端站点域名配置

### 11.5 密钥与配置管理

所有敏感配置应由服务端或平台环境变量管理，包括：

- 模型平台 API Key（DeepSeek、通义、豆包、ChatGPT）
- 后端服务访问密钥
- Supabase 高权限凭证（service_role key）
- 埋点写入密钥
- 邮件服务 API Key
- 线索通知收件人邮箱配置

前端仅持有公开接入所需的最小配置。

## 12. 安全、权限与风控边界

### 12.1 基本原则

整站安全与权限设计遵循以下原则：

- 身份可信
- 规则集中
- 敏感能力后置
- 前端不裁决业务权限

### 12.2 注册与测试资格

- 未注册用户可填写测试表单
- 只有完成 Supabase Auth 邮箱注册后才可正式发起测试
- 邮箱验证异步进行，不阻塞测试功能使用
- 后端必须校验身份（验证 Supabase JWT），不依赖前端本地状态作最终裁决

### 12.3 免费次数规则

- 每个邮箱账号默认赠送 3 次免费测试机会
- 只有测试成功完成后才扣减一次
- 免费次数以后端计算为准
- 次数检查与扣减需在同一业务事务中完成，避免并发穿透

### 12.4 模型调用控制

所有模型调用必须经过后端统一授权和限流：

- 未注册用户不可触发模型调用
- 免费次数用尽不可继续调用
- 前端不得直连目标模型平台

### 12.5 防滥用要求

首期架构应预留基础防滥用能力：

- 前端防重复点击
- 测试接口限流
- 注册接口限流
- 同一用户短时重复提交控制
- 结果详情归属校验

### 12.6 埋点与隐私边界

- 埋点数据应与敏感业务数据最小必要隔离
- 手机号、原始回答等敏感数据不应无控制进入分析面板
- 行为分析链路不应直接暴露完整注册信息或测试原文

## 13. 扩展性与演进边界

### 13.1 当前架构的扩展方向

当前架构已为以下能力预留边界：

- 更多平台支持
- 多轮测试与多模板采样
- 品牌别名与英文名识别
- citation / 来源分析
- 历史测试列表
- 用户测试中心
- 内容配置后台
- 销售线索流转
- 更精细的漏斗分析

### 13.2 后续最可能独立演进的部分

后续最可能从轻实现升级为独立能力的模块包括：

- 模型执行层
- 配置管理层
- 埋点分析层
- 销售线索承接层

### 13.3 演进约束

后续扩展应优先沿现有边界演进，而不是继续把新逻辑堆叠在前端页面层。

要保持可扩展性，需持续遵守：

- 前端不承担核心业务计算
- 业务规则不散落在多个端
- 配置集中管理
- 埋点命名与业务口径统一
- 业务事实数据与分析事件数据分离

## 14. 监控与可观测性最低要求

首期整站应具备以下最低可观测能力：

- 前端页面访问与错误监控
- 后端接口调用日志
- 模型调用失败日志
- 关键转化事件埋点监控

说明：

- 首期不要求建设完整 observability 平台
- 但必须具备基础排障与漏斗分析能力

## 16. MVP 验收标准

当以下条件满足时，可认为整站技术架构满足首期要求：

- 前端站点（Next.js + Vercel）、业务后端（Python + FastAPI）、Supabase 与埋点层边界清晰
- 测试页和结果页都可展示总体累计数据
- 前端仅直接接入 Supabase Auth 认证能力，不直接承担核心业务写入
- 核心测试与免费次数规则由后端统一控制
- 跨域（CORS）配置正确，前端可正常访问业务后端 API
- 埋点体系覆盖首页到后续联系动作的主要漏斗链路
- 开发、测试和生产环境在逻辑上明确隔离
- 敏感密钥不暴露到前端
- "联系销售"动作可触发线索邮件通知
- 可通过后台页面或 Supabase Dashboard 查看用户数据
- 架构已为后续多平台、多轮测试和销售承接预留边界

## 17. 结论

首期 GEO 官网整站应采用”前端站点（Next.js + Vercel）独立、业务后端（Python + FastAPI）独立、Supabase 中台承接邮箱认证与数据、埋点体系正式纳入范围”的分层技术架构。

该架构能够在保持 MVP 实现效率的前提下，支撑官网内容承接、注册留资、免费测试、结果展示和后续转化，并通过清晰的系统边界、数据域划分和漏斗埋点，为后续功能扩展和增长优化提供稳定基础。
